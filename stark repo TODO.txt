Architecture
------------
	* Connectivity slices? (maybe I can already work with it)
		At the moment I have to choose one material model. I can't have different cloth models in the same scene.
		Changing materials would require recompilation.
		This problem could be solved by having different connectivity arrays, one per material. But that would be insane in terms of combinations with everthing else.
		This could be the solution to the collision combinatorial problem!!
		A solution could be connectivity slices:
			-> Use the lambdas to the pointer and size to point to differet ranges of the same array.
		This allows to use the same array for different energies (materials).
		
		The problem of the combinatorial explosion is that is inverse: we want different connectivities concatenated into one.
		Should I support multiple connectitivity pointers natively in symx? What about the data?
			The way things work now, we need contiguous in memory conn and data!

	
	* Drop the old Energy and PhysicalSystem inheritance-based interface in favor of lambdas. Old problems:
		- Physical systems with multiple dofs or no dofs
		- Energies are naturally attached to the data. Unnenecessary complexity and incompatibility when splitting.
		
		The new system allows to set lambdas as they are needed without enforcing anything. List of lambdas:
			.add_callback_before_time_step();
			.add_callback_before_energy_evaluation();
			.add_callback_max_allowed_step();
			.add_callback_is_state_valid();
			.add_callback_after_energy_evaluation();
			.add_callback_after_time_step();
			.add_callback_write_frame();
